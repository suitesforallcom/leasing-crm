# SCALING_PLAN.md — масштабирование SuitesForAll до 50–100 операторов

> Живой документ. Источник истины по инициативе «убрать sync-конфликты при росте числа операторов».
> Статус обновляется по фазам. Started 2026-05-30.

## Проблема (корень)

Весь воркспейс хранится в ОДНОМ Firestore-документе `workspaces/default/data/state`,
и каждая правка переписывает его целиком под оптимистичной блокировкой (`_rev` + транзакция).

Firestore сериализует конкурентность **на уровне документа** → все писатели стоят в одной
очереди к одному замку. Две стены, обе из монолита:

1. **Конкуренция записи.** P(конфликт) ≈ частота × длительность записи. 804 KB пишутся ~1–2 сек —
   окно огромное; с ростом писателей конфликт → 100%. 50–100 операторов один документ не просериализует.
2. **Жёсткий лимит 1 MB/документ.** Сейчас ~804 KB = 80%, растёт. Стена, не зависящая от конкуренции.

Оба решаются одним: **разобрать монолит на документ-на-сущность.**

## Целевая архитектура (document-per-entity)

```
workspaces/{ws}/
  meta/state                         крошечный индекс (глоб. _rev, флаги)
  settings/global                    редкие записи
  buildings/{bId}                    мета здания (редко)
  units/{unitId}                     ГОРЯЧАЯ сущность: операторы правят юниты
  units/{unitId}/payments/{ym}       каждый платёж — отдельный док
  leases/{leaseId}  tenants/{id}  contracts/{id}
  ledger/{autoId}                    append-only журнал денег (добавление НЕ конфликтует)
  ingest/{gmail|calendar|aircall}/…  шумные серверные потоки, вне горячего пути
```

Эффект:
- Операторы на РАЗНЫХ юнитах → разные документы → ноль конфликтов. Конфликт остаётся только
  при реальной правке ОДНОГО юнита двумя людьми (и это правильно — его и надо ловить).
- `cloud-function` (биллинг) пишет в `payments/{ym}` и `ledger` — не задевает юнит оператора.
- Платежи (~195 KB, самый жирный/записываемый кусок) уезжают из горячего дока → стена 1 MB исчезает.
- Деньги — append-only ledger (совпадает с `FINANCIAL_INVARIANTS.md`).

**Ключ к выполнимости:** приложение по-прежнему собирает тот же in-memory `state` объект — только
из стримов коллекций, а не из одного дока. Весь рендер/агрегации (Rent Roll, P&L, дашборды)
работают как есть. Меняется только слой персистентности.

## Repository/DAO-слой (линчпин)

Перед всеми мутациями — тонкий слой:
```
repo.updateUnit(id, patch)     repo.setPayment(unitId, ym, rec)
repo.appendLedger(entry)       repo.updateLease(id, patch)  …
```
UI зовёт высокоуровневые мутаторы; КАК они пишут (старый монолит / новые доки) решает флаг
`settings.syncV2`. Меняем бэкенд, не трогая тысячи мест вызова.

## Миграция — strangler-fig, по кускам, обратимо

Каждая фаза: за флагом → dual-write (пишем в оба места) → сверочный job сравнивает старое/новое →
переключаем чтение → soak → удаляем старое. Откат на любом шаге = вернуть флаг.

- [~] **Фаза 0 — фундамент.** Флаг `settings.syncV2` (off). Read-only замер базовой метрики
  (`sfaSyncStats()` — разбивка размера по секциям). DAO-seam (по мере фаз). *← в работе*
- [ ] **Фаза 1 — платежи (наибольший выигрыш, делаем первым).** `u.payments[ym]` →
  `units/{id}/payments/{ym}`. Dual-write + сверка. Убирает львиную долю 804 KB и самый горячий
  путь записи (биллинг) из монолита. ~80% боли уходит здесь.
- [ ] **Фаза 2 — юниты** в отдельные доки. После неё операторы на разных юнитах = ноль конфликтов.
- [ ] **Фаза 3 — остальное:** buildings/floors-мета, leases, tenants, contracts, settings, ingest.
- [ ] **Фаза 4 — снос монолита:** старый `data/state` → крошечный индекс или удаляется; dual-write убирается.
- [ ] **Фаза 5 — закалка денег:** ledger как источник истины для «оплачено»; идемпотентные ключи
  на каждую Stripe/auto-invoice запись (частично уже есть).

## Требует одобрения Tony (gated, по каждой — отдельный запрос)

- `firestore.rules` — новые пути с той же ролевой моделью.
- `firestore.indexes.json` — composite/collectionGroup индексы (напр. все платежи для Rent Roll).
- `functions/index.js` — биллинг пишет в `payments`/`ledger`, не в state.
- Схема `state.*` — собственно декомпозиция.

## Риск/сроки

- Недели работы, но фазы независимы; Фаза 1 одна снимает большую часть проблемы.
- Риск под контролем: dual-write + автосверка + флаг + существующие бэкапы/снапшоты (`daily-snapshots.js`).
- НЕ делаем: big-bang, смену БД, CRDT/OT (избыточно).
